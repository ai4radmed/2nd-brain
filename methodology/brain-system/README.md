# brain-system — 이 볼트 자체에 관한 문서

이 폴더는 second-brain 시스템을 **시스템으로서 관리**하기 위한 메타 영역이다. 볼트 운영 규칙, 철학, 네이밍 규약, 의사결정 기록을 여기에 둔다.

권위 문서는 vault 루트의 `CLAUDE.md` (각 사용자가 직접 작성). 이 README 는 인간 가독용 상세 설명.

---

## second-brain 이란 무엇인가

Tiago Forte 의 *Building a Second Brain* (BASB) 방법론을 기반으로 한 개인 지식관리 시스템. 외부에서 들어오는 지식·정보·생각을 "두 번째 뇌" 역할의 시스템에 축적·조직·연결하여, 인간 뇌의 휘발성·편향·용량 제약을 보완한다.

---

## 두 폴더, 하나의 시스템

```
2nd-brain/
├── knowledge/              ← 생각의 집 (Obsidian 볼트)
│   - .md 마크다운만
│   - 요약·생각·연결
│   - 양방향 링크의 그래프
│
└── sources/                ← 원본의 집 (The Sources)
    - PDF·docx·xlsx·이미지
    - 불변하는 참조 자료
    - 외부에서 들어온 증거
```

두 폴더는 **동일한 PARA 구조**로 미러링되어 있어, 한쪽의 경로로 다른 쪽 위치를 예측할 수 있다.

### 왜 분리하는가

**도구의 강점을 살리기 위해**: Obsidian 은 텍스트·링크에 최적화, 파일시스템·클라우드는 바이너리에 최적화. 섞으면 둘 다 느려진다.

**원본의 불가침성**: PDF 는 한 번 저장하면 수정하지 않는다. 변경·주석·생각은 MD 에서. 원본이 깨끗하게 보존된다.

**검색 최적화**: "그때 읽었던 논문이 뭐였지?" 는 Obsidian 검색(요약 있음). "그 PDF 원본 어디 있지?" 는 파일 탐색기 검색(폴더 구조). 각자 잘하는 일.

**인식론적 대립**: `knowledge` = 내가 소화·체계화한 것, `sources` = 외부에서 온 원본 증거. 이 구분이 **어디에 무엇을 둘지** 판단을 쉽게 만든다.

## PARA 방법론

Tiago Forte 의 BASB 에서 제안된 분류 체계. 주제가 아닌 **실행 가능성(actionability)** 을 기준으로 분류한다.

- **P — Projects**: 마감이 있는 능동 목표. 끝나면 Archive 로.
  예: "2026 논문 X 작성", "5월 학회 발표"

- **A — Areas**: 지속적 책임 영역. 마감 없음.
  예: "진료", "연구", "학회 활동", "재정", "건강"

- **R — Resources**: 관심 주제별 참고자료.
  예: "PET imaging", "AI in medical imaging", "리더십"

- **A — Archive**: 완료된 모든 것. 삭제하지 않고 여기로.

### 왜 PARA 인가

주제 분류는 끝이 없다 (논문 주제, 환자군, 질환, 기법…). 반면 역할·프로젝트·관심사는 상대적으로 안정적이다. "지금 중요한가" 를 기준으로 하니 **뭘 볼지 결정이 빠르다**.

최상위 4개 + 각각 소수의 하위로 관리 가능. 수백 개 폴더 트리로 폭발하지 않는다.

### 5폴더 prefix 패턴

실무 운영에서는 `00_inbox/` 를 추가해 5폴더로 운영하기를 권장:

- `00_inbox/` — 분류 전 임시 보관. 주간 리뷰에서 비움.
- `01_projects/` — Projects (마감 있는 능동 목표)
- `02_areas/` — Areas (지속 책임 영역)
- `03_resources/` — Resources (주제별 참고)
- `04_archive/` — Archive (완료·비활성)

번호 prefix 가 파일 탐색기·Obsidian 의 알파벳 정렬 시 자연스러운 우선순위를 만든다.

## 브레인화 (Brainify)

외부 정보를 second-brain 시스템에 편입시키는 행위를 **"브레인화"** 라고 부른다. BASB 의 CODE 프레임워크 중 **Capture → Organize → Distill** 단계를 포괄하는 한국어 동사. Express 는 별도 활동(글쓰기·발표)으로 분리.

| 라벨(예: Gmail) | 의미 | 조치 |
|---|---|---|
| `브레인화/완료` | 흡수 완료, 동반 노트 생성됨 | Archive |
| `브레인화/보류` | 외부 파일 다운로드 대기 | Inbox 유지 |
| `브레인화/중복` | 중복 참고용 보존 | Archive |
| `브레인화/불필요` | 뉴스레터·광고 등 | Archive |

세부 워크플로우는 별도 문서로 분리될 예정 (Gmail·Contacts·외부 파일 staging 패턴 등).

> **운영 원칙**: 브레인화는 건별 사람 승인 없이 **자동**으로 하고, 품질은 **주 1회 감사**로 모은다 — [자동 우선·주간 감사](./automation-weekly-audit.md).

## 동반 노트 (Companion Note) 패턴

원본 PDF 등을 참조할 때, MD 노트가 지식 그래프의 **허브 노드** 가 된다.

```
      [노트 B] ←→ [논문A.md] ←→ [노트 C]
                      ↓ (file:// 링크)
                 [논문A.pdf]  ← 외부, 그래프 바깥
```

PDF 는 그래프에 직접 참여하지 못한다 (파일 포맷이 링크 메타데이터를 담지 못함). 대신 MD 가 PDF 를 대리하여 네트워크에 참여한다.

### 왜 이게 우아한가

- **완전한 양방향 링크 유지** — 볼트 내 MD 끼리는 정상 백링크
- **그래프 뷰 온전** — MD 가 허브 노드 역할
- **원본 PDF 깔끔하게 외부** — 볼트가 가볍고 빠름
- **지식 단위 추출** — PDF 100 페이지 중 중요한 몇 문장이 MD 에 추출됨

### 템플릿

```markdown
---
title: 논문 또는 자료 제목
source: 출처 (URL, 저자, 기관 등)
date: YYYY-MM-DD
tags: [tag1, tag2]
sources: <원본 파일 URI>
---

# 제목

> [원본 열기](<원본 파일 URI>)

## 한 줄 요약

## 핵심 내용
- ...

## 내 생각

## 관련 노트
- [[관련 노트 1]]
- [[관련 노트 2]]
```

`<원본 파일 URI>` 는 운영 환경에 따라 다름:
- WSL2 native + Obsidian on Windows → `file://wsl.localhost/Ubuntu/home/<user>/projects/2nd-brain-vault/sources/...`
- 컨테이너 안 또는 WSL2 native 셸 (동일 상대 경로) → `~/projects/2nd-brain-vault/sources/...`
- vault 상대 경로 (Obsidian 위키링크) → `[[sources/.../파일]]`

### 파일명 규칙

- **이벤트·캡처**: `YYYY-MM-DD_출처_내용.확장자`
- **학술 논문**: `저자_연도_주제.확장자`
- 공백 금지(언더바·하이픈). 최상위 폴더는 영문, 하위 파일명은 한글 허용.

## Claude 와의 협업

vault 루트의 `CLAUDE.md` 를 Claude Code 가 세션 시작 시 자동으로 읽는다. 거기에 시스템 핵심 규약·브레인화 원칙·제약을 압축해 두면 새 세션에서 "오늘 인박스 브레인화해줘" 라고만 해도 적절한 폴더에 파일을 생성할 수 있다.

### 지침 계층과 토큰 예산

Claude Code 가 읽는 지침은 7개 층(system → enterprise → user → auto memory → project → local → subdir)으로 쌓이고, 집중력 유지를 위해 사용자 통제 지침(CLAUDE.md + memory) 합계는 **5–10K 토큰** 권장. 상세는 [[claude-instruction-layers]].

### 시스템 진화 방법

규약이나 원칙을 바꾸고 싶을 때:
1. 이 README 또는 별도 메모에 결정 근거를 먼저 적음
2. vault 루트의 `CLAUDE.md` 를 업데이트
3. 다음 Claude 세션에서 새 규약이 자동 반영됨

## 운영 규칙

### 민감정보 분리 (도메인별 최우선)

- **의료**: PHI(환자건강정보)는 절대 일반 클라우드·개인 계정에 들어가서는 안 됨. 폐쇄망에만. 임상 사례는 익명화 후 저장 또는 저장 금지.
- **법무·재무**: 클라이언트 식별 정보는 별도 보안 영역.
- **일반**: vault 가 평문 마크다운으로 외부 클라우드(Drive·Dropbox 등)에 동기화될 가능성을 가정하고 작성.

각자 도메인 규약·법률(개인정보보호법·HIPAA 등)을 사전에 점검할 것.

### 다중 기기 동기

볼트 정본 위치와 기기 간 동기 전략(클라우드 미러링·Syncthing·git push 등)은 사용자 환경에 따라 다름. 어떤 방식이든 다음을 지킨다:

1. **동시 편집 회피** — 같은 파일을 두 기기에서 동시에 열지 않음. Obsidian 은 한 번에 한 기기.
2. **기기 전환 전 동기 완료 확인** — 동기화 도구의 상태 표시 확인.
3. **정기 백업** — 동기화 ≠ 백업. 월 1회 별도 위치에 zip/tar 백업 권장.

### 주간 리뷰 (30분, 매주 고정)

매주 정해진 시간(예: 금요일 오후·일요일 저녁)에:

1. `00_inbox/` 의 각 항목을 3초 내 판단 → 적절한 PARA 폴더로 이동
2. Tier 판단:
   - 자주 볼 것 → 요약 노트 추가
   - 한 번 보고 끝 → 단순 분류만
   - Probably never → Archive 또는 삭제
3. `01_projects/` 점검: 진행/멈춤/새로 시작
4. 30일 이상 건드리지 않은 Project → Area 또는 Archive 이동 검토

이 30분이 시스템의 엔진. 없으면 Inbox 가 쌓여서 붕괴한다.
